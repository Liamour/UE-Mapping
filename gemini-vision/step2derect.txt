一、 蓝图内部逻辑提取与 AI 语义逆向 (对应需求 1 & 2)1. 物理层可行性（C++ 引擎端）：完全可行，但必须采用“按需惰性加载（Lazy-Load）”策略。当前陷阱： 绝对不能在全局扫描时提取所有蓝图的内部节点。提取内部连线需要使用 LoadObject<UBlueprint>() 将资产完全加载到内存并解析其 UEdGraph。全局执行会导致 UE5 编辑器内存溢出或直接假死。架构重构（按需深潜扫描）： * 维持 AssetRegistry 的全局浅层扫描作为基础拓扑 。当用户在前端 React Flow 中点击某个 SystemNode （或触发“深度审计”指令）时，通过 JSON-RPC 向 C++ 桥接层发送目标路径。C++ 层仅针对该特定蓝图进行深度反向工程：遍历 UbergraphPages（事件图表）和 FunctionGraphs，提取关键的 UEdGraphNode（如 Branch、Cast、Custom Events、RPC 调用）及其引脚（Pins）连线关系。将这些杂乱的图表数据序列化为精简的 JSON AST（抽象语法树）并回传前端。2. 认知层可行性（AI 逻辑重构）：将 LLM 作为“语义反编译器”。单纯展示内部节点连线对人类来说依然是“面条代码”。数据流转： 前端接收到 C++ 传来的内部节点 JSON 后，直接送入 Volcengine LLM。Prompt 工程（关键节点识别）： 要求大模型提取核心业务骨架。例如，让 LLM 忽略基础的数学运算节点，只抓取“状态机切换”、“网络 RPC 同步”、“关键变量修改”的节点。UI 映射 (TE 工业美学)： 在右侧 Sidebar 中，新增 “深层微观架构 (Micro-Architecture)” 面板。不再使用复杂的连线图，而是将 AI 总结出的业务逻辑通过高对比度的 Timeline（时间轴）或 Step-by-Step 列表展示，关键变量或警告使用我们标志性的霓虹橙（#ff6600）高亮。二、 HITL (Human-in-the-Loop) 专家级人工干预 (对应需求 3)可行性：100% 前端掌控，完全契合当前基于 Zustand 的状态管理 。AI 不是万能的，高级 OPC 开发者必须拥有对数据的绝对修改权。这属于状态管理与交互层面的升级。1. 节点自定义编辑 (Node Mutability)交互逻辑： 双击 SystemNode 或 ClusterGroupNode  的标题区域，直接切换为 <textarea> 状态（延续我们之前为解决 CEF 拼音输入法拦截所做的物理规避策略）。状态同步： 修改后的内容通过 Zustand store 实时更新图表状态 ，并在导出时固化到 JSON 中 。2. 战术标签系统 (Tactical Tagging)数据结构： 在节点的数据模型中新增 tags: string[] 字段。UI 呈现： 在节点下方或侧边栏增加“圆角胶囊”样式的 Tag 容器（采用深空灰 #151515 配合内阴影模拟物理凹槽）。允许用户手动输入标签，或从 AI 推荐的语义标签库中快速点选。3. 动态物理重组 (Dynamic Clustering)交互逻辑： 允许用户打破 AI 的分类。通过 React Flow 的原生事件，支持拖拽 SystemNode 离开当前的 ClusterGroupNode ，或者将其拖入另一个自定义的聚类盒子中。底层逻辑： 监听拖拽释放事件，动态修改子节点的 parentId，触发 Zustand 重新渲染 。三、 首席架构师的后续执行推演 (Next Steps)如果要吃下这些功能，我们需要分为两个 Sprint 来执行，确保系统不崩溃。Sprint 1: 前端赋权与专家干预（见效最快，成本最低）前端动作： 在 SystemNode 和 ClusterGroupNode 组件中加入“编辑模式” 。右侧边栏重构： 在 Sidebar 增加详情面板 ，支持编辑描述、增删 Tags。状态流转： 完善 Zustand 中的 updateNodeData 和 reparentNodes 方法 。Sprint 2: 引擎深潜与 AI 反编译（核心技术壁垒）C++ 动作： 在 AICartographerBridge 中新增 RPC 方法 DeepScanBlueprint(FString AssetPath) 。编写逻辑读取 UEdGraph 并提取精简版节点拓扑。LLM 动作： 设计专门的 prompt，将 C++ 传出的 AST JSON 翻译为人类可读的业务逻辑（例如：“此蓝图接收输入，判断是否在地面，然后触发 Jump RPC”）。UI 动作： 将 AI 的总结结果注入侧边栏的“蓝图内部构建逻辑”面板。
