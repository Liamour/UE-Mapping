# 火山引擎请求30秒超时Bug报告
## 基本信息
| 项 | 详情 |
|----|------|
| 错误类型 | 请求超时 |
| 错误描述 | Request Timeout (30s). The proxy or network is blocking the request. |
| 触发逻辑 | AbortController 30秒超时阈值被触发，主动终止无响应请求 |
| 影响范围 | AI聚类请求完全无响应，无法返回结果 |

---

## 根因分析（按排查优先级排序）
### 1. 最高概率：本地网络/代理拦截（70%）
- 本地开启了VPN/全局代理软件，Vite代理的出口流量被代理劫持，无法访问火山引擎`ark.cn-beijing.volces.com`域名
- 公司/内网防火墙/安全策略拦截了对火山引擎域名的HTTPS请求，导致流量被丢弃无响应
### 2. 次高概率：IP白名单限制（20%）
当前使用的火山方舟Endpoint绑定了IP白名单，当前机器的公网出口IP不在白名单内，请求被火山引擎安全策略拦截丢弃，无任何返回
### 3. 低概率：模型推理耗时过长（8%）
当前项目节点数量过多（55个），大模型处理时间超过30秒，触发超时阈值，属于业务侧正常超时
### 4. 极低概率：Vite代理/网络配置错误（2%）
- Vite代理的target域名配置错误/网络不通，无法转发请求
- 本地DNS解析异常，无法解析火山引擎域名
---

## 排查定位步骤
### 第一步：判断请求是否到达Vite代理层
查看运行`npm run dev`的PowerShell终端：
- **如果没有打印`🚀 [Vite Proxy] Sending request to Volcengine...`日志**：说明请求没到代理层，是前端UI逻辑/页面缓存问题，强制刷新页面重试即可
- **如果只打印了🚀请求日志，没有打印✅响应日志**：说明Vite代理已经转发请求，但没有收到火山引擎返回，问题出在Vite到火山的网络链路，请继续往下排查
- **如果打印了✅响应日志还是超时**：说明大模型处理耗时超过30秒，需要调高超时阈值
### 第二步：验证外部网络连通性
在本地浏览器直接打开火山引擎地址`https://ark.cn-beijing.volces.com/api/v3/models`，看是否能正常访问返回JSON：
- 如果可以正常返回：说明本机到火山的网络正常，是Vite代理配置问题
- 如果无法访问/超时：说明本机网络/策略拦截了请求，是网络环境问题
---

## 修复方向建议
### 临时解决方案（快速验证）
1. 关闭本地所有VPN/代理软件，重试请求
2. 切换到手机热点等不受公司/内网策略限制的网络，重试请求
3. 如果是IP白名单问题，将当前公网IP添加到火山方舟Endpoint的IP白名单中
### 代码层面优化
如果模型推理确实耗时过长，可将`llmService.ts`中的超时阈值从30秒调整为60秒：
```typescript
const timeoutId = setTimeout(() => controller.abort(), 60000); // 从30000改为60000
```
### 代理方案优化
如果是公司网络必须走代理，可修改Vite代理配置，给代理加上`secure: false`和自定义代理服务器配置。
